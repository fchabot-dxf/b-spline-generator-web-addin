# CAM Builder — Context Sheet

Living reference for what the CAM Builder add-in is, what it does, what's
currently working / broken, and the verified API patterns it relies on.
Read this first when picking up the project after a break, or when handing
context to a new LLM session.

Companion docs:
- `CAM_API_NOTES.md` — verified Fusion CAM API patterns (timing, enum
  strings, post-add idiom)
- Runtime audits of real Fusion documents live in `cam-builder-cam-debug.log`

---

## What this is

A Fusion 360 add-in inside the `bspline-frame-builder` suite that takes a
parametric Design (b-spline panel + surrounding frame + stock placeholder)
and stamps out the **Manufacturing Models and Setups** needed to machine
it on a 3-axis router. Operations (toolpaths) are NOT generated by the
add-in — the user adds those manually in the CAM workspace once the
Setups exist. The add-in's job is: *get me to a clean, correctly-bound,
correctly-oriented set of Setups in two clicks instead of fifty*.

Output target: g-code for an Ultimate Bee CNC running a DDCS Expert 1.1
controller (M350 firmware), via Fusion's standard post-processor with
DDCS-specific tweaks. Manual tool change workflow (see the `ddcs-expert`
skill for macro patterns).

---

## What I'm building (the parts the CAM Builder works on)

A decorative wood panel with a curved face. The Design has three logical
body groups:

- **Stock blank** — a placeholder rectangular box sized to the raw
  material I'll feed in. Not a real machined body; just a reference.
- **B-spline panel set** — the curved panel itself. May have a "Stamped"
  variant (post-tooling) and a "Clean" variant (pre-tooling); CAM only
  cares about the Stamped one if it exists, otherwise Clean. Surfaces in
  the panel set are reference-only — strip them before machining.
- **Frame** — the rectangular surround that holds the panel. Machined
  flat in its own setup.

Two-sided machining for the b-spline:
1. Machine the **back** of the panel (flat side, gets a pocket/profile).
2. Flip the part, machine the **top** of the panel (curved face,
   3D-finish toolpath).

Same WCS corner for both b-spline sides; the second side is just
`flipY=true` from the first.

## Pipeline

```
  Design tree
    │
    ▼
  cam_workspace.activate_cam()       (switch to Manufacture workspace)
    │
    ▼
  cam_engine.mm_builder.build_all_mms()
    │   ── creates 3 ManufacturingModels: stock, bspline_set, frame
    │   ── each starts as a full snapshot of the Design, then
    │      _apply_occurrence_filter() deletes occurrences per rule
    │   ── stock MM gets _populate_stock_placeholder() (parametric box)
    │   ── bspline_set MM gets _apply_bspline_panel_filter()
    │      (Stamped > Clean, strip surfaces)
    │   ── all MMs get _propagate_user_parameters_to_mm()
    │
    ▼
  cam_engine.setup_builder.build_all_setups()
    │   ── creates 4 Setups: Stock, B-spline Back, B-spline Top, Frame
    │   ── each Setup binds to the matching MM via setupInput.models
    │   ── stockMode set via TYPED enum (adsk.cam.SetupStockModes.*)
    │   ── WCS configured via parameter dict on the LIVE setup
    │
    ▼
  User adds toolpath operations manually in the CAM workspace
```

Top-level orchestrator: `cam_engine.cam_coordinator`. Entry point /
command handler: `cam-builder.py` (palette UI lives in `ui/html/`).

## Setup table (point of truth)

| Setup           | MM rule       | Stock intent      | WCS origin   | WCS orient | Box point | flipY |
|-----------------|---------------|-------------------|--------------|------------|-----------|-------|
| Stock           | `stock`       | auto_bbox         | model_origin | model      | —         | false |
| B-spline Back   | `bspline_set` | auto_bbox         | box_point    | axesXY     | `top 1`   | false |
| B-spline Top    | `bspline_set` | from_prev_setup   | box_point    | axesXY     | `top 1`   | true  |
| Frame           | `frame`       | auto_bbox         | model_origin | model      | —         | false |

The two B-spline setups are **identical** except for stock intent (Top
inherits from Back) and `flipY`. Same MM, same corner, same axes.

`SETUP_SPECS` in `cam_engine/setup_builder.py` is the single source of
truth — declarative table, edit there to add/change setups.

## Body-filter rules (mm_builder)

`_DELETE_OCC_CLASSES` in `cam_engine/mm_builder.py`:

| Rule          | Deleted occurrence classes                          | What's left                |
|---------------|-----------------------------------------------------|----------------------------|
| `stock`       | panel, frame, stock_placeholder, **unknown**        | nothing (placeholder built)|
| `bspline_set` | frame, stock_placeholder                            | panel + unknown wrapper    |
| `frame`       | panel, stock_placeholder                            | frame + unknown wrapper    |

Why DELETE rules instead of KEEP: when Fusion creates an MM from a
Design, it wraps everything under a single 'unknown' occurrence named
after the document. KEEP-style rules would nuke that wrapper and take
all the descendants with it. DELETE rules let the wrapper ride along.

The **stock** rule is the exception — it nukes the wrapper too, then
`_populate_stock_placeholder` builds a fresh parametric box.

## Verified API patterns (the gotchas)

### Timing: parameters AFTER add()
CAM Setup parameters' `.choices` enum is empty on `setupInput.parameters`
— it only populates once the Setup is materialised. Always do:
```
setup = cam.setups.add(setupInput)
# ... write parameters on setup.parameters, NOT setupInput.parameters
```
(Verified from Autodesk's `SetViseOriginAsSetupWCSOrigin` and
`CreateSetupsFromHoleRecognition` samples.)

### Stock mode: typed enum > parameter dict
```python
setup.stockMode = adsk.cam.SetupStockModes.RelativeBoxStock
```
The `job_stockMode` parameter-dict path silently rejects `'relativebox'`
on current builds. Typed enum sidesteps the whole issue. Param dict
still works for `'fixedbox'`, `'fromsolid'`, `'previoussetup'` — used as
fallback in `_set_stock_mode()`.

### WCS axes are swapped
The `wcs_orientation_axisX` slot expects what naive intuition would call
the Y direction reference, and vice versa. We bind `yConstructionAxis`
to `wcs_orientation_axisX` and `xConstructionAxis` to
`wcs_orientation_axisY`. Verified empirically against the runtime
result.

### Manufacture workspace `activeProduct` quirk
Inside the Manufacture workspace, `app.activeProduct` returns a
`CAMProduct`, not a `Design`. To get the source design when you're
already in Manufacture:
```python
doc = app.activeDocument
ds = doc.products.itemByProductType('DesignProductType')
design = adsk.fusion.Design.cast(ds)
```
Fixed in `setup_builder._get_origin_axes`. Still latent in
`mm_builder.py` (see Open Issues).

### Verified parameter strings (single-string, not lists)
- `wcs_origin_mode`: `'modelOrigin'` (model_origin), `'stockPoint'`
  (box_point)
- `wcs_orientation_mode`: `'modelOrientation'` (model), `'axesXY'`
  (select_x_y)
- `wcs_origin_boxPoint`: `'top 1'` (verified from audit JSON)
- `wcs_orientation_flipY`: expression `'true'` / `'false'` (string, not
  Python bool)

The candidate-list framing in `parameter_introspect.py` is now mostly
redundant — the verified strings are single values. Slated for refactor.

## Box point map (incomplete — empirical)

The `wcs_origin_boxPoint` parameter takes a corner-or-center identifier
like `'top 1'` or `'top center'`. Confirmed values from audits of real
setups:

| Identifier      | Meaning                              |
|-----------------|--------------------------------------|
| `'top 1'`       | top face, corner 1 (used by both b-spline setups) |
| `'top center'`  | top face, center                     |

Other values (`'top 2'`, `'top 3'`, `'top 4'`, `'bottom 1'` etc.) are
plausible by pattern but not yet verified. Set casing-fallback in
`set_choice` tries `'top 1'`, `'top1'`, `'top_1'` to be safe across
Fusion builds.

## Hardware target

- **Machine**: Ultimate Bee CNC (BulkMan3D), ~1m × 1m work area
- **Controller**: DDCS Expert 1.1 (M350 firmware)
- **Tool changes**: manual, via macros — see `ddcs-expert` skill
- **Post-processor**: Autodesk Fusion's standard, with DDCS-specific
  tweaks for G53 (verified syntax), no G10, no G28

## Open issues / regressions

- **Frame MM ends up bodiless** — confirmed via 2026-04-28 18:31 run.
  The MM filter is doing the right thing (kept 'Clean', 'Stamped',
  '(Unsaved)'; deleted 'B-Spline Set' [class=panel]), but the iteration
  order of `allOccurrences` (Clean → Stamped → B-Spline Set → (Unsaved))
  reveals that **Clean and Stamped are children of B-Spline Set**. When
  the frame rule deletes the parent, the children are dragged along,
  leaving only the empty `(Unsaved)` wrapper. **The user's Design has
  no separate `Frame` component.**

  Two paths forward:
    1. Author a real Frame component in the source Design (sibling of
       B-Spline Set under the wrapper).
    2. Add a `_populate_frame_geometry()` to `mm_builder` analogous to
       `_populate_stock_placeholder()` — the CAM Builder builds the
       frame parametrically, the same way it builds the stock blank.

- **`PARAM PROP / STOCK PLACEHOLDER` warnings — partially addressed.**
  `cam_utils.get_design()` now centralises the
  `activeProduct` → `itemByProductType('DesignProductType')` fallback.
  - `setup_builder._get_origin_axes` uses it (clean drop-in).
  - `mm_builder._propagate_user_parameters_to_mm` uses it as a fallback
    after the original `Design.cast(app.activeProduct)` attempt. Note:
    this writes into the *source* Design, not the MM's derived design,
    but the existing-name guard makes it idempotent for source params
    (which is the common case — we're snapshotting FROM the source).
  - `mm_builder._populate_stock_placeholder` does **not** use the
    fallback — falling back to the source Design would create the stock
    body inside the user's actual design (pollution). It still skips
    cleanly with an improved warning message. The proper fix here is to
    bind stock geometry to the Setup directly via `setup.stockSolids`
    instead of routing through the MM (TODO).

- **Operations not generated**. By design — user adds them manually.
  Future work could template common operations (3D Adaptive on the
  curved face, parallel finish, contour around the frame).

## Refactor surface (ranked, deferred)

1. **Drop dead candidate strings** in
   `parameter_introspect.STOCK_MODE_CANDIDATES['auto_bbox']` (every
   string rejected; typed enum is the only working path).
2. **Dedupe `_log` shim** — currently in both `setup_builder.py` and
   `parameter_introspect.py`; `cam_utils/cam_logger.py` exists but
   isn't used.
3. **Extract param-I/O helpers** (`_set_entity_param`,
   `_set_bool_param`, `_set_stock_mode`, `_get_origin_axes`) from
   `setup_builder.py` into `parameter_introspect.py` so it becomes the
   one true API surface for "writing things to a Setup".
4. **Centralize the activeProduct fallback** in `cam_utils.get_design()`
   (also fixes the latent `mm_builder` bug).
5. **Split `mm_builder.py`** (794 lines) into `mm_filter.py` (per-rule
   body deletion) and `mm_stock.py` (the bbox-and-extrude flow for the
   Stock MM). Defer until Pass 3 (frame reorient + parametric stock
   extrude) is implemented — that work will reshape the file anyway.

`SETUP_SPECS` is good as-is. `cam_coordinator.py` and
`cam_workspace.py` are right-sized.
